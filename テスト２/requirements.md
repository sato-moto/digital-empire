## 1. プロダクト概要

### 1.1 ビジョン

> 「最もUI/UXが洗練された日本麻雀アプリとして、一人のクリエイターが運営するにもかかわらず、業界トップクラスの収益を生み出すプラットフォームを構築する」

麻雀アプリ市場は既に成熟しているが、UI/UXの陳腐化・広告過多による体験劣化が蔓延している。
本プロダクトはクリエイター自身のブランド・コンテンツ発信力を武器に、
「遊びたくなるUI」「続けたくなる設計」「課金したくなる価値提供」を三位一体で実現する。

### 1.2 プロダクト名（仮）

**「天鳳道（テンホウドウ）」** ※仮称。ネーミングは別途ブランディング検討

### 1.3 ターゲットユーザー

| セグメント | 属性 | ニーズ | 収益貢献度 |
|-----------|------|--------|-----------|
| **コア層** | 20〜40代・麻雀経験者・スマホゲームユーザー | 本格ルール・快適な操作 | 高（課金メイン） |
| **ライト層** | 麻雀初心者〜中級者 | 学習・暇つぶし | 中（広告＋小課金） |
| **競技層** | 雀荘通い・大会参加者 | 統計・段位・対戦 | 最高（プレミアム課金） |
| **配信者層** | ゲーム実況・麻雀YouTuber | 見栄えするUI・共有機能 | 中（UGC流入） |

### 1.4 差別化ポイント

| 競合 | 課題 | 本アプリの解決策 |
|------|------|----------------|
| 雀魂 / 天鳳 | UI古め・PC寄り | モバイルファーストの洗練されたUI |
| MJモバイル | 広告過多・課金圧が強い | 体験を壊さない課金設計（選択式） |
| ジャラン | ルール限定 | フル日本ルール対応＋カスタムルール |
| 一般麻雀アプリ | クリエイター性ゼロ | クリエイター自身のキャラ・ストーリー |

### 1.5 プロダクトの独自価値（UVP）

- **クリエイターIPとの融合**: アプリ内にクリエイター自身のキャラクター・ナレーション・限定コンテンツを組み込み、ファンが課金する動機を生む
- **AI対戦の質**: 強さレベルを細かく調整できるAI。初心者向け解説モード搭載
- **統計ダッシュボード**: 牌効率・アガリ率・放銃率など詳細な成績分析（差別化機能）
- **クリエイターコラボ**: 他クリエイターとのコラボ牌デザイン・テーブルテーマ販売

## 2. マネタイズ戦略

### 2.1 収益モデル全体像

```
収益構造（フェーズ2以降の目標比率）

  └ サブスクリプション  25%
  └ コスメ課金          15%
  └ ガチャ/くじ         5%
広告収益（IAA）          30%
  └ 動画リワード        20%
  └ インタースティシャル 10%
```

## 3. 機能要件

### 3.1 コア機能（麻雀ゲームロジック）

#### 3.1.1 対局モード

| モード | 説明 | フェーズ |
|--------|------|---------|
| 一人用 vs AI | 1人〜3人のAI対戦。強さ5段階 | MVP |
| フリー対局 | ルール自由設定での練習 | MVP |
| 段位戦（ランク戦） | ポイント制・段位昇降 | Phase2 |
| リアルタイム対人 | マルチプレイヤー（最大4人） | Phase2 |
| チュートリアル対局 | 初心者向け解説付き | MVP |
| 日替わり道場 | 特定局面からのパズル形式 | Phase2 |
| 大会・トーナメント | 期間限定イベント対局 | Phase3 |

#### 3.1.2 対応ルール

**必須ルール（MVP）**:
- 四人麻雀（東風戦・半荘戦）
- 基本役: リーチ・タンヤオ・ピンフ・メンゼンツモ・一盃口・三色同順・一気通貫・七対子・対々和・三暗刻・混老頭・清老頭・国士無双・九蓮宝燈等（全40役以上）
- ドラ・裏ドラ・赤ドラ（0.5枚設定可）
- フリテン・振り聴ルール
- 途中流局（4枚カン・9種9牌等）
- 連荘・本場点

**拡張ルール（Phase2）**:
- 三人麻雀
- カスタムルール保存・共有
- ローカルルール設定（一発・カンドラ等のON/OFF）

#### 3.1.3 AIエンジン要件

| レベル | 強さ感 | 実装方法 |
|--------|--------|---------|
| Lv1（入門） | 初心者と同程度 | ルールベースAI（牌効率なし） |
| Lv2（初級） | 基本牌効率 | ルールベースAI（シャンテン計算） |
| Lv3（中級） | 安全牌意識・鳴き判断 | ルールベースAI＋統計テーブル |
| Lv4（上級） | 状況判断・読み | 強化学習 or モンテカルロ木探索 |
| Lv5（最強） | 天鳳位相当 | ML モデル（Phase2以降） |

#### 3.1.4 判定ロジック要件

- シャンテン数計算（全形式対応）
- 有効牌計算・待ちの表示
- 点数計算（符・翻計算・役満）
- フリテン判定（自分・同順・リーチ後）
- 和了可能判定（誤ロン防止）

### 3.2 UI/UX機能

#### 3.2.1 操作仕様

- タップ操作でツモ・打牌・鳴き・リーチ
- スワイプでのソート（手牌整列）
- ダブルタップ確認（誤操作防止。設定でOFF可）
- 操作タイムアウト（AI代打オプション）
- 横画面・縦画面両対応

#### 3.2.2 アシスト機能

| 機能 | 説明 | 課金要否 |
|------|------|---------|
| シャンテン表示 | 現在のシャンテン数をリアルタイム表示 | 無料 |
| 有効牌ハイライト | ツモ切り以外の選択肢をハイライト | スタンダード以上 |
| 危険牌警告 | 放銃リスクの高い牌を色表示 | プレミアム |
| 自動ソート | ツモ後の手牌自動整列 | 無料（ON/OFF） |
| 一発・裏ドラ確認 | 和了時の自動計算・表示 | 無料 |

### 3.3 ソーシャル・エンゲージメント機能

#### 3.3.1 段位・ランクシステム

```
段位設計（雀力の可視化）
────────────────────────
初段 → 二段 → ... → 十段
  └ 昇段: 段位ポイント蓄積
  └ 降段: 下限あり（気持ちよく続けられる設計）
  └ 段位ボーナス: 各段位達成でコスメ付与（継続動機）

称号システム
────────────────────────
全国段位ランキング（月間・週間）
達成称号（特定条件で解放: 1000局達成・国士和了等）
```

#### 3.3.2 実績・バッジシステム

- 対局実績（局数・アガリ数・役満達成等）60種以上
- バッジをプロフィールに表示
- SNS共有機能（実績達成時・和了結果カード）

#### 3.3.3 クリエイター連動機能

| 機能 | 内容 | マネタイズ効果 |
|------|------|-------------|
| クリエイター限定コンテンツ | オリジナルキャラ・ボイス・背景 | ファン課金動機 |
| クリエイターコメント | 対局中にランダム出現するオリジナル台詞 | エンゲージメント向上 |
| クリエイター配信連携 | YouTubeライブ・TikTokと連動した応援機能 | 流入獲得 |
| ファンランキング | 累積課金額によるファンランク表示 | 課金競争促進 |
| 限定イベント通知 | クリエイターからのプッシュ通知 | リテンション向上 |

#### 3.3.4 フレンド・コミュニティ機能（Phase2）

- フレンド登録・プロフィール閲覧
- フレンド招待対局
- ギルド / 卓グループ機能（最大20人）
- チャット（スタンプのみ・テキストは Phase3）

### 3.4 課金・ショップ機能

#### 3.4.1 ショップ画面要件

- 牌石購入フロー（Apple/Google課金 IAP）
- アイテム一覧（カテゴリ分類・プレビュー付き）
- 期間限定表示（カウントダウンタイマー）
- おすすめバッジ・初回限定バッジ
- 購入履歴・所持アイテム管理
- プレゼント機能（フレンドへのアイテム贈与、Phase3）

#### 3.4.2 サブスクリプション管理

- プラン選択・変更・解約フロー
- 継続日数カウント・ボーナス表示
- 解約防止オファー（休止オプション提示）
- レシート検証（サーバーサイド必須）

#### 3.4.3 広告管理

- リワード広告視聴フロー（任意・非強制）
- 広告上限管理（日次カウント）
- 広告ネットワーク切り替え設定（ASO最適化）

## 4. 非機能要件

### 4.1 パフォーマンス要件

| 項目 | 要件 |
|------|------|
| アプリ起動時間 | コールドスタート 3秒以内 |
| 画面遷移 | 300ms以内 |
| 対局応答（AI思考） | Lv1〜3: 1秒以内 / Lv4〜5: 3秒以内 |
| フレームレート | 60fps維持（対局中アニメーション） |
| メモリ使用量 | 300MB以下（iOS/Android中低スペック機対応） |
| オフライン対応 | AI対戦はオフライン動作 |
| APKサイズ | 初回DL 100MB以下 |

### 4.2 セキュリティ要件

| 項目 | 要件 |
|------|------|
| 通信 | HTTPS/TLS 1.3 必須 |
| 認証 | JWT + リフレッシュトークン |
| 課金検証 | サーバーサイドレシート検証（Apple/Google API）|
| 不正対策 | チート・改ざん検知（対人戦） |
| データ保護 | パスワード不保持（OAuth2 ソーシャルログイン）|
| GDPR/個人情報 | 利用規約・プライバシーポリシー整備 |
| 課金不正 | チャージバック検知・アカウント停止フロー |
| レート制限 | API エンドポイントにリクエスト制限 |

### 4.3 スケーラビリティ要件

| 項目 | 要件 |
|------|------|
| 同時接続 | Phase2: 1,000人 / Phase3: 10,000人 |
| データベース | 水平スケール対応設計 |
| CDN | アセット配信はCDN経由 |
| マイクロサービス対応 | コア機能とソーシャル機能を分離可能な設計 |
| ゲームサーバー | WebSocket接続のステートレス化 |

### 4.4 可用性要件

| 項目 | 要件 |
|------|------|
| SLA | 99.5%以上（Phase2以降 99.9%目標） |
| メンテナンス | 事前告知 / 深夜帯 / 最小30分 |
| 障害復旧 | RTO: 30分以内 / RPO: 1時間以内 |
| バックアップ | 日次フルバックアップ + 時間毎増分 |

### 4.5 対応環境

| プラットフォーム | バージョン |
|--------------|----------|
| iOS | 15.0以上 |
| Android | 9.0（API 28）以上 |
| Web（PWA） | Chrome / Safari 最新2バージョン（Phase2）|
| 画面サイズ | 4.7インチ〜タブレット対応 |

### 4.6 アクセシビリティ

- 色覚多様性対応（赤緑色覚異常向けの牌デザイン選択肢）
- フォントサイズ変更対応
- 振動フィードバック（ON/OFF選択可）

## 5. 技術スタック候補

### 5.1 プラットフォーム選定

**推奨: React Native（Expo管理ワークフロー）**

| 観点 | React Native | Flutter | Unity | ネイティブ |
|------|-------------|---------|-------|----------|
| 開発速度 | 高 | 高 | 中 | 低 |
| 一人開発の負荷 | 低 | 低 | 中 | 高 |
| UI品質 | 高 | 高 | 中（2D） | 最高 |
| ゲームロジック実装 | 可 | 可 | 最適 | 可 |
| 広告SDK対応 | 良好 | 良好 | 良好 | 良好 |
| Web展開 | 容易（RN Web）| 可（Flutter Web）| 難 | 不可 |
| 採用理由 | **JSエコシステム活用・高速プロトタイプ** | | | |

> Unity も選択肢だが、カジュアルゲーム向けの広告/課金SDK連携の容易さからReact Nativeを推奨。
> ゲームロジックの重い処理はWebAssembly（Rust）に切り出す設計を推奨。

### 5.2 フロントエンド

| 用途 | 技術 |
|------|------|
| メインフレームワーク | React Native（Expo SDK 51+） |
| 状態管理 | Zustand（軽量・一人開発向き） |
| アニメーション | React Native Reanimated 3 |
| ナビゲーション | Expo Router（ファイルベース） |
| ゲームロジック | TypeScript + WASM（Rust） |
| グラフィック | React Native Skia（牌アニメーション） |

### 5.3 バックエンド

| 用途 | 技術 | 理由 |
|------|------|------|
| APIサーバー | Hono（Cloudflare Workers） | エッジ実行・低コスト・スケール自動 |
| リアルタイム対戦 | Cloudflare Durable Objects | WebSocketステート管理 |
| データベース | Turso（LibSQL / SQLite互換） | 一人運営コスト最小・エッジ展開 |
| 認証 | Clerk or Supabase Auth | ソーシャルログイン設定容易 |
| ファイルストレージ | Cloudflare R2 | コスト最小・CDN統合 |
| プッシュ通知 | Expo Push Notifications | 無料・SDK統合済み |
| 分析 | Mixpanel（行動分析）+ Sentry（エラー）| 無料枠あり |

### 5.4 マネタイズ関連SDK

| 機能 | SDK |
|------|-----|
| iOS課金（IAP） | StoreKit 2（RevenueCat経由推奨） |
| Android課金 | Google Play Billing（RevenueCat経由） |

## 9. リスクと対策

### 9.1 事業リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| ストア審査リジェクト（課金・ガチャ）| 高 | 中 | ガイドライン事前確認・確率明示・弁護士確認 |
| 競合大手の新機能投入 | 高 | 高 | クリエイターIP差別化の徹底・コミュニティ深耕 |
| 収益未達（課金転換率未達）| 高 | 中 | MVPでA/Bテスト・価格帯調整・初回オファー強化 |
| 一人運営による開発遅延 | 中 | 高 | フェーズ分割・OSSコンポーネント活用・外注部分明確化 |
| クリエイター活動低下による流入減 | 高 | 低 | アプリ自体のSEO・口コミ設計・自走コミュニティ構築 |

### 9.2 技術リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| リアルタイム対戦の同期ズレ | 高 | 中 | Cloudflare Durable Objectsでサーバー権威型実装 |
| 課金二重付与バグ | 最高 | 低 | 冪等性設計・サーバーサイド検証・トランザクション管理 |
| アプリサイズ肥大化（コスメ増加）| 中 | 高 | CDN動的配信・On-demand asset loading |
| AI強さのバランス崩壊 | 中 | 中 | 統計データによる継続チューニング・フィードバック収集 |
| Cloudflare Workers制限（CPU/メモリ）| 中 | 低 | 重い処理はバッチ分離・DO使い分け設計 |

### 9.3 法務・コンプライアンスリスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| ガチャ規制強化 | 高 | 中 | 確率表示徹底・コンプガチャ回避・弁護士顧問契約 |
| 資金決済法（仮想通貨扱い）| 高 | 低 | 牌石の失効設計・未使用残高上限5万円以内の設計 |
| 著作権（麻雀牌デザイン）| 中 | 低 | オリジナルデザイン制作・デザイナー契約での権利帰属明確化 |
| 未成年課金トラブル | 高 | 中 | 月額上限設定・保護者確認フロー・年齢確認 |
| 個人情報保護法・GDPR | 中 | 低 | プライバシーポリシー整備・データ最小化設計 |

### 9.4 運営リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| チート・不正プレイ（対人戦）| 高 | 中 | サーバー権威型ゲームロジック・異常値検知 |
