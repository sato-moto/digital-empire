# 外部API失敗時のトリアージ・ルール（自動適用）

このルールは、外部API（Gemini / Google AI / OpenAI 等）を呼ぶスキル・スクリプトでエラーが発生した際に自動適用される。
対象スキル: slide-generator / create-course / thumbnail-designer / その他 API を使う全スキル

---

## 絶対原則：「キー無効」と独断で結論してはならない

API呼び出しが失敗しても、即「APIキーが無効」と結論してはならない。
**必ず以下のトリアージを完了させてから報告せよ**。

### 失敗時に即やること（順序厳守）

1. **HTTPステータスの確認**
   - `400` = リクエスト不正（キー無効 OR モデル名違い OR リクエスト形式誤り）
   - `401` = 認証失敗（キー無効の可能性が高い）
   - `403` = 権限不足（課金/有効化未完了）
   - `404` = エンドポイント/モデル名不在（キーは関係ない）
   - `429` = レート制限（一時的、リトライで解決）
   - `500`系 = サーバ側障害

2. **ステータスと「キー無効」メッセージだけで判断しない**
   - `400` の body に `"API_KEY_INVALID"` が入っていても、同一キーで別モデルが通れば「モデル指定の問題」
   - **必ずエンドポイント/モデル名を本番スクリプトと照合する**

3. **テストスクリプトを書く場合**
   - **本番スクリプトが叩くのと同一のモデル名・エンドポイント・認証方式**を使うこと
   - 異なるモデル名で404を受けて「キー無効」と結論するのは厳禁

---

## 複数候補の .env がある場合

- プロジェクト内に複数の `.env` が存在する場合、**全件を同一テスト条件で並列検証**すること
- 1つが失敗したら他も失敗と仮定してはならない（キーがローテーションされている可能性）
- 検証スクリプトは**キー本体を絶対にログ/標準出力に出さない**（セキュリティ・プロトコル準拠）

---

## 「APIが使えない」を理由に成果物を縮退させる判断

API失敗を理由に、成果物を本来より低品質なバージョン（例：図解なしテキスト版PPTX）に切り替える判断は、**事前承認の対象外**である。

- 独断で縮退版を出力することは**禁止**
- 必ずマスターに以下を報告してから判断を仰ぐ：
  1. どのAPIが何回失敗したか
  2. HTTPステータスとエラーメッセージ（キー本体は出さない）
  3. 縮退版で走るか、原因調査を優先するか、マスターに選ばせる

## 本ルールの策定経緯

2026-04-24、MATH-M-01-B-a / B-b 制作時、`generate_diagrams.js` が 400 INVALID_ARGUMENT を返したため、
Claude が「全 .env のキーが無効」と独断で結論し、図解なしのテキスト版PPTXで完成させて報告した事案をきっかけに制定。

実際には：
- 本番スクリプトが使う画像生成モデル `gemini-3-pro-image-preview` の制約による一時的な400だった
- 同じキーで同モデルを再呼び出ししたら通った（レート制限の可能性）
- テストスクリプトは別モデル `gemini-2.0-flash` を叩いて全件404を受け、二重に誤判定していた

マスターから「同じこと起こさんようにルール化しとけよ」と明確な訓戒を受け、本ルールを制定。
