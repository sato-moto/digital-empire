# プロジェクトルール

## コミュニケーション
- ユーザーとのやり取りは日本語で行うこと
- エラーメッセージやログも日本語で出力すること

## 🤖 意図の自動判定とスキル・エージェントの自動実行

ユーザーが「記事作って」「動画作って」などとカジュアルに依頼してきた場合、
コマンドを思い出させることなく、以下のルールに従って**Skillを自動で選択・実行**すること。

### 記事生成（`article-creator` スキルを使用）

以下のいずれかに該当する場合は **`create-article` スキル**を自動実行する：

| ユーザーの発言例 | 判定キーワード |
|---|---|
| 「記事作って」 | 記事、note、stand.fm、ブログ |
| 「〇〇について書いて」 | 書いて、まとめて、コンテンツ作って |
| 「〇〇の記事お願い」 | 記事、投稿、コンテンツ |

- テーマが発言内に含まれていればそのままスキルに渡す
- テーマが不明な場合のみ1回だけ確認する

### 動画制作（`create-video` スキルを使用）

以下のいずれかに該当する場合は **`create-video` スキル**を自動実行する：

| ユーザーの発言例 | 判定キーワード |
|---|---|
| 「動画作って」 | 動画、ビデオ、Remotion |
| 「〇〇の動画お願い」 | 動画、映像、ショート |
| 「動画化して」 | 動画化、映像化 |

### リサーチ・市場調査（`researcher` エージェントを使用）

以下のいずれかに該当する場合は **`researcher` エージェント**を自動起動する：

| ユーザーの発言例 | 判定キーワード |
|---|---|
| 「〇〇を調査して」 | 調査、リサーチ、分析、競合 |
| 「スクレイピングして」 | スクレイピング、データ取得、ランキング取得 |
| 「市場を調べて」 | 市場、トレンド、人気記事、ランキング |

- 対象URL/キーワードが含まれていればそのまま渡す
- 取得件数が不明な場合はデフォルト300件で進める

### HTMLビジュアライゼーション

以下のいずれかに該当する場合はHTMLファイル生成を行う：

| ユーザーの発言例 | 出力スタイル |
|---|---|
| 「HTMLで見せて」「可視化して」 | analysis.html（白背景モダン） |
| 「ランキングHTML」 | ranking.html（ダーク系カード） |
| 「レポート作って」「分析して」 | analysis.html（コンサルレポート風） |
| 「地方メディア記事」「熊取町の記事タイトル」 | local-media-strategy.html（地方メディア転用戦略） |

### 地方メディア記事戦略

「地方メディア」「熊取町の記事」「kumatori-info.com 向け」などのキーワードが含まれる場合：
- 市場データと地方メディアの文脈を**ブリッジマトリクス**で接続する
- タイトル案には必ず「対象読者・想定媒体・SEOキーワード候補・選定理由」を付与する
- 想定媒体は kumatori-info.com / note / stand.fm の3媒体から選ぶ
- 地名（熊取町・大阪・泉南）を含むロングテールKWを必ず1つ以上入れる
- 出力HTML: `local-media-strategy.html`（白背景・コンサルレポート風）

### KW型タイトル戦略

「KW」「キーワード」「タイトル案」「型を指定して」などが含まれる場合：
- 指定されたKW型（例：初心者訴求型・メディアジャンル型・爆速テーマ型）に分類して提案する
- 各案に必ず付与する：対象読者・想定媒体・選定理由（参照した市場データの型）・SEOキーワード候補
- フィルタリングボタン付きのHTMLで型別に切り替えられるようにする
- 出力HTML: `kw-strategy.html`（白背景・3カラム型提案カード）

**KW型の定義：**

| 型名 | 説明 | 典型フレーズ |
|-----|------|------------|
| 初心者訴求型 | 参入障壁を下げ購買母数を最大化 | 「ゼロから」「未経験でも」「知識なし」「超初心者でも」 |
| メディアジャンル型 | プラットフォーム名×AIで検索意図を直接キャッチ | 「YouTube×AI」「TikTok×」「Kindle×AI」「インスタ×」 |
| 爆速テーマ型 | 時間コストの低さを先出しして即買い衝動を引き出す | 「20分で」「激時短」「今日から」「ビフォーアフター数字」 |
| 地方メディア型 | 地名×AI×体験記で競合が薄いロングテールを狙う | 「熊取町」「地方メディア」「1人で」「地域創生」 |

### 判定の優先順位
1. 「動画」ワードが含まれる → `create-video`
2. 「記事」「書いて」「note」「stand.fm」ワードが含まれる → `create-article`
3. 「調査」「スクレイピング」「リサーチ」「ランキング取得」ワードが含まれる → `researcher`
4. 判定が難しい場合のみ「記事・動画・リサーチ、どれですか？」と1回確認する

## コーディング
- Node.js CommonJS形式（`require` / `module.exports`）を使用
- Remotionコンポーネントはパスカルケース（例: `VideoComposition`, `SubtitleOverlay`）
- 変数・関数はキャメルケース（例: `generateScript`, `audioPath`）
- ファイル名はケバブケース（例: `video-creator.js`, `style-template.js`）

## エラーハンドリング
- API呼び出しは必ず try-catch で囲む
- エラー発生時はユーザーにわかりやすい日本語メッセージを表示
- リトライ可能なエラー（API rate limitなど）は自動リトライを実装

## 動画制作
- 動画制作開始前に必ずユーザーに目的・スタイルを確認する
- ユーザーが素材（台本・画像・音声）を提供した場合はそれを優先的に使用
- 完成動画には通し番号を付与し、一覧HTMLで管理する
- BGMが指定されない場合はデフォルトBGMを使用する

## 記事生成

### 出力形式
- 記事は必ず `.md`（Markdownファイル）として保存する
- 保存先：`C:\Users\motok\OneDrive\Desktop\Claude code\yuniko\[テーマ名]記事.md`
- コードブロック（```markdown）でのチャット出力とファイル保存を併用する

### スタイル基準（ユニコ記事スタイル）
- 参考記事フォルダ：`参考記事/` 内のファイルをスタイル参照に使用する
- 構成：体験談 → 現状解説 → ツール紹介 → 地域活用 → FAQ → まとめ
- 見出し：番号付き（`## 1.` / `### 1-1.`）
- AI感のある表現を避ける（「〜することが重要です」「〜が求められます」など禁止）
- 熊取町要素はデフォルトで含める（ユーザーから省略指示があった場合のみ外す）

### ハッシュタグ
- 記事生成時は必ずnote用・stand.fm用の2種類を生成・末尾に追加する
- note用：最大15個（`#note` を末尾に含める）
- stand.fm用：最大8個に厳選（`#standfm` `#ラジオ` を含める）

### stand.fm説明文
- 記事生成時は必ず150文字前後のstand.fm放送説明文も生成する
- 熊取町メディア運営者の文脈を含め、記事の要点を簡潔にまとめる
