# 実践動画 収録指示書 ガイドライン（汎用）

本ガイドラインは、講座内の「実践（実務）パート」において**スライドを使わず、画面収録映像のみで構成する講義**を作成する際の、**収録指示書の作り方**を定義する。

対象：講師（撮影者）。本ドキュメントに従って指示書を出力すれば、講師は迷わず画面収録できる状態にする。

---

## 1. 基本思想

### 1-1. 実践パートはスライド不要、画面収録映像で完結させる
- 実務系の講義は「概念説明」ではなく「手の動かし方」を伝えるもの。
- スライドを作って差し込むより、**講師の画面操作 + ナレーション**だけで完結させたほうが、視聴者の理解と再現性が高い。
- よって実践パートの成果物は次の3点セットとする：
  1. **指示書（Markdown）**：講師が読んで撮影準備〜本番まで遂行できる手順書
  2. **指示書（CSV）**：操作と台本を1行ずつ対応させた撮影用カンペ（2カラム）
  3. **収録された画面録画動画**（撮影後）

### 1-2. 「操作」と「セリフ」を1対1で対応させる
- 撮影現場で迷う最大の原因は「今どこの操作をしながら、何を喋ればいいか」が分からないこと。
- そのため、**操作内容と台本（読み上げ用セリフ）は同一行に必ず併記する**。

### 1-3. プロンプト・入力文字列は一言一句記載する
- 「〜について聞く」のような方針記述は禁止。
- ChatGPTやNotionへの入力は、**実際にコピペできる文字列をそのまま記載**する。
- 入力欄の場所、押すボタン、開くタブ、ファイル名まで具体に書く。

## 2. 指示書（Markdown）のテンプレート

実践動画 1本 = Markdownファイル1つ。ファイル名は `practice-{講座番号}-{連番}.md`。

```markdown
# 実践動画 指示書：{講座番号} - {タイトル}

## 0. メタ情報
- **講座**: {コース名 / 講座番号 / 講座タイトル}
- **対応章**: {How章のどのパートか}
- **想定尺**: {例: 5〜7分}
- **使用ツール**: {例: ChatGPT (GPT-5), Googleスプレッドシート}
- **アウトプット**: {視聴者が動画を見終えた後に手元に残るもの}

## 1. 収録の目的（このシーンで視聴者に持ち帰らせるもの）
- {1〜3行で明確に}

## 2. 事前準備（撮影前にすべて完了させること）

### 2-1. 環境（Windows 固定）

**収録は Windows 環境で行う。** 本ガイドライン全体を通して Windows 表記に統一する（Mac 併記禁止。詳細は §3-6 参照）。

- **画面解像度**: 1920×1080（フルHD）に設定
- **デスクトップ**: タスクバー / 通知センター / デスクトップアイコンが**映らない**ようにする
  - タスクバー自動非表示 ON（タスクバー右クリック → タスクバー設定 → 「タスクバーを自動的に隠す」ON）
- **通知**: 集中モード ON（Slack, メール, Teams等の通知を完全停止）
- **ブラウザ**: シークレットウィンドウ推奨（ブックマークバー・拡張機能アイコンを非表示）
- **ウィンドウサイズ**: 録画範囲いっぱいに最大化

### 2-2. 開いておくタブ・アプリ
- {例: ChatGPTのトップ画面（新規チャット状態）}
- {例: 空のGoogleスプレッドシート（ファイル名: 「{コース名}_実践_{講座番号}」）}
- 不要なタブはすべて閉じる

### 2-3. ダミーデータ
- 個人情報・他社情報・社内情報は一切使わない
- 例示用の架空データを事前に用意：{具体例}

### 2-4. 映してはいけないもの（NG事項）
- デスクトップバー / タスクバー
- ブックマークバー、拡張機能アイコン
- メールアドレス、社名、顧客名、契約情報
- 他のアプリの通知ポップアップ
- 別タブの中身（業務情報の映り込み）

## 3. 操作 × 台本（本編）

> 詳細は同梱の CSV 指示書（`practice-{講座番号}-{連番}.csv`）を参照。
> Markdown側では **操作のフェーズごと** に区切り、補足が必要な部分のみ説明を加える。

### Phase 1：{フェーズ名 / 例：プロンプト設計}
- **狙い**: {このフェーズで視聴者に何を見せるか}
- **操作の流れ**:
  1. {操作の要約}
  2. {操作の要約}
- **補足**: {画面外で講師が意識すべきこと、よくあるつまずき}

### Phase 2：{フェーズ名}
（同様）

## 4. 収録時の話し方ルール
- 1文を短く（40字以内目安）
- 操作の **直前** に「次に〜します」と予告 → 操作 → 「これで〜になりました」と結果確認、の3拍子
- 入力中の沈黙はOK。早送り編集を前提に、入力完了後に必ず1拍置く
- 専門用語を出したら必ず1度だけ言い換える

## 5. 撮影後チェックリスト
- [ ] NG事項（タスクバー等）が映っていないか全編確認
- [ ] 通知ポップアップの映り込みなし
- [ ] 個人情報・他社情報の映り込みなし
- [ ] 音声クリア、無音区間が長すぎない
- [ ] 想定尺の±30%以内に収まっている
```

## 3. 指示書（CSV）のフォーマット

### 3-1. カラム定義（2カラム固定）

| カラム名 | 内容 |
|---|---|
| `操作内容` | 講師が画面上で行う操作を、**そのまま実行できる粒度**で記述。開く画面・クリックする場所・入力する文字列（プロンプト等は一言一句）・押すボタンを明示 |
| `台本` | その操作をしている**最中に読み上げるセリフ**。1行=1ショット感覚。話し言葉、40字以内目安 |

### 3-2. ファイルエンコーディング規定（文字化け防止 SSOT）

**目的:** Windows / Mac 両環境、Excel / エディタ / ブラウザ、いずれで開いても文字化けしない。本ガイドラインがエンコーディング仕様の Single Source of Truth（SSOT）。他ファイル（`output-naming.md` 等）はここを参照するだけで、個別に別定義を持たない。

#### ファイル種別ごとのエンコーディング表

| ファイル種別 | エンコーディング | 改行コード | 備考 |
|---|---|---|---|
| `practice-{N}.md` / `-final.md` | UTF-8 **without** BOM | LF | Markdown 標準。BOM ありだと textlint など一部パーサが崩れる |
| `practice-{N}.csv` / `-final.csv` | UTF-8 **with** BOM | CRLF | Excel で直接開いても文字化けしないため BOM 必須 |
| `practice-{N}.xlsx` / `-final.xlsx` | (Excel 内部形式) | — | `/xlsx` スキルが openpyxl で生成。UTF-8 内部処理 |
| `practice-{N}-guide-win.pdf` / `-mac.pdf` | (PDF 内部形式) | — | Noto Sans JP フォント埋め込み（`generate_practice_pdf.py` 標準） |

#### CSV の詳細仕様

- ファイル名: `practice-{講座番号}-{連番}.csv`
- 区切り: カンマ
- ヘッダ行: `操作内容,台本`
- セル内改行: `\n`（LF）でエスケープし、セル値全体は `"..."` で囲む。Excel / `/xlsx` スキル側ではセル内改行として正しく表示される

#### 禁止事項（文字化けの直接原因）

- ❌ CSV を UTF-8 without BOM で保存（Excel で日本語が文字化け）
- ❌ CSV をシフトJIS / CP932 で保存（macOS で文字化け・絵文字で破損）
- ❌ MD に UTF-8 with BOM を付与（textlint などの Markdown パーサでエラー）
- ❌ PDF 生成時に Noto Sans JP 以外のフォント指定（フォントない環境で豆腐化）
- ❌ Excel 生成時にセル内改行を `\n` 文字列のまま書き込む（改行として解釈されない）

### 3-3. 記述粒度の基準（重要）

**NG例**：
```
操作内容: ChatGPTに課題発見のプロンプトを入力する
台本: ChatGPTを使って課題を整理してみましょう
```
→ 何を入力するのか不明。これでは撮影できない。

**OK例**：
```
操作内容: ChatGPTのトップ画面で、入力欄に以下を一言一句そのまま貼り付けてEnterを押す。\n---\nあなたは事業企画の専門家です。以下の企業の課題を、3C分析のフレームに沿って5つ洗い出してください。\n企業：架空の中堅SaaS企業「ExampleWorks」\n事業：中小企業向けの勤怠管理SaaS\n出力形式：箇条書き、各項目は50字以内
台本: それでは実際にChatGPTを使ってみます。今回はこちらのプロンプトを入れます。事業企画の専門家として、3C分析のフレームに沿って課題を5つ出してもらう、という指示です。
```

ポイント：
- **画面の場所**（どこをクリック / どこに入力）
- **入力文字列の全文**（コピペできる形）
- **次のアクション**（Enter / 送信ボタン / タブ切替）
を必ず操作内容に含める。

### 3-4. CSVサンプル（抜粋）

```csv
操作内容,台本

```

### 3-5. 台本の語り口ルール（最重要）

実践動画は**ナレーション付き解説映像ではなく、1本の講義動画**として成立させる必要がある。台本は「操作の説明文」ではなく「**講師が画面の向こうの受講者に語りかけるセリフ**」として書くこと。

#### 前提
- 講師は**アバター**である。カメラに向かって話す演出（会釈・カメラ目線への切り替え・カメラ目線で〜等）は使わない。すべて画面（操作対象）を映したまま語る
- 動画の開始時点で**使用ツールの画面はすでに開いた状態**で映っている。最初の操作までは画面を見せたまま語りで導入する
- オープニングで「ようこそ」「〇本目の講座へ」等の歓迎演出は使わない。**「ここからは実践に入っていきます」のように、実践パートに入ったことが自然に伝わる切り出し**にする
- 前講座の参照は「**前回までに〜を学んでいただきましたね**」のニュートラルな接続にする。「教養編で」「実務編1本目」など**講座カテゴリ名や講座番号を出さない**（コース構造を意識させない）

#### 必須構成要素

| # | 要素 | 入れる場所 | 役割 |
|---|---|---|---|
| 1 | **実践への切り出し** | CSV1行目 | 「ここからは実践に入っていきます」等。歓迎挨拶ではなく、フェーズ転換の宣言 |
| 2 | **前講座との接続** | 切り出し直後 | 「前回〇〇を学んでいただきましたね」と既習内容を想起させる |
| 3 | **今回のゴール宣言** | オープニング内 | この動画を見終わると何ができるようになるかを明示 |
| 4 | **Phase間のブリッジ** | 各Phase冒頭 | 「Step1ができたので、次は〜に進みます」と段差を作らない |
| 5 | **受講者への声かけ** | 操作の合間に随時 | 「ここは迷いやすいので一緒に考えましょう」「みなさんも手元で同じようにやってみてください」等 |
| 6 | **間（ま）と相づち** | 適宜 | 「いいですね」「ここポイントです」等、無人感を消す |
| 7 | **振り返り** | エンディング前 | 今日やったことを3〜4点で要約 |
| 8 | **ねぎらい** | エンディング | 「お疲れさまでした」「最後までご覧いただきありがとうございました」 |
| 9 | **次への接続** | 末尾 | 「次の講座では〜をやっていきます」と次アクションを示す |

#### 語り口の原則
- **「です・ます」調を全文で徹底**する。1文でも常体（〜だ／〜である／言い切り）が混ざると講義の質感が崩れる
- **二人称は「みなさん」**（「あなた」「視聴者の方」はNG）
- **一人称は「私」**（「講師」「筆者」はNG）
- **体言止めの連発禁止**。振り返りパートで「Step1で〜、Step2で〜」と並べるときも「〜し、〜してもらい、〜する流れでしたね」のように動詞で締める
- **くだけた接続詞・口語の置換**：
  - 「だから」→「なので／だからこそ」
  - 「でも」→「ただ／ですが」
  - 「〜けど」→「〜ですが」
  - 「〜じゃなくて」→「〜ではなく」
  - 「〜から（理由）」言い切り → 「〜ので／〜ためです」
- 文末で「〜ない。」「〜がある。」と常体で言い切らない。必ず「〜ありません／〜あります」まで書く

#### 敬語チェック観点（執筆後に必ず通す）
- [ ] 全文末が「です・ます／ません・ました／ましょう」のいずれかで終わっているか
- [ ] 「〜だ」「〜である」「〜だろう」が混ざっていないか
- [ ] 「だから／でも／〜けど／〜じゃなくて」のくだけた接続が残っていないか
- [ ] 体言止めが3行以上連続していないか
- [ ] みなさんへの呼びかけが2〜3行に1回以上入っているか
- 操作の説明と感情の言葉を**混ぜる**：「では入力します。……はい、出ましたね。これがポイントです」
- **問いかけ**を入れる：「ここ、なぜだと思いますか？」（答えてから先に進む）
- **共感**を入れる：「ここは私も最初つまずいたところです」

#### NG例 / OK例

**NG（説明文）**：
```
台本: ChatGPTを開いて新規チャットを作成し、プロンプトを入力する。
```

**OK（講義）**：
```
台本: それでは早速やっていきましょう。まずChatGPTを開いて、新しいチャットを立ち上げます。今日使うプロンプトはこちらです。一緒に見ていきましょう。
```

#### オープニング・エンディングの最小テンプレ

**オープニング（CSVの最初2〜3行に分割して入れる。操作はせず、開いている画面を見せたまま語る）**
> 「ここからは実践に入っていきます。前回までに{前講座のテーマ}を学んでいただきましたね。これを実際のケースで使い倒していきます。ゴールは{今回のゴール}を作ることです。流れは{N}ステップ。{流れの要約}。それでは始めていきます。」

**エンディング（CSVの最後2〜3行に分割して入れる。最終成果物の画面を映したまま語る）**
> 「今日のデモは以上です。振り返ると、{3〜4点の要約}でした。{最重要ポイントの再強調}。お疲れさまでした。次の講座では{次のテーマ}に進みます。それでは、また次の動画でお会いしましょう。」

### 3-6. OS 表記ポリシー（重要・Windows 固定）

**CSV および Markdown 指示書は Windows 表記で固定する。両 OS 併記は禁止。**

#### なぜ Windows 固定か

- CSV は講師が動画収録する際の**カンペ（台本）**である。
- 講師の撮影環境は **Windows 固定**（画面収録動画が Windows 画面のため）。
- Mac ショートカットが台本に混ざると、講師は読み上げ中に混乱し、動画と台本の不一致が発生する。
- 受講者向けの OS 分岐は、**PDF 生成時（Step 6.5-P）**で自動実施される。収録指示書（CSV/MD）側では一切考慮しない。

#### 表記ルール

- **キーボードショートカット**: Windows 表記のみ（`Ctrl + S`、`Windowsキー + 左矢印` 等）
- **ウィンドウ操作**: Windows 表記のみ（`Windowsキー + 左矢印で画面を左半分に固定する` 等）
- **アプリ名**: Windows 版のアプリ名（エクスプローラー、メモ帳、タスクマネージャー 等）
- **ファイルパス**: Windows 形式（`C:\Users\...`、バックスラッシュ区切り）
- **環境設定**: Windows の設定名（タスクバー自動非表示、集中モード 等）

#### 禁止パターン

❌ `Ctrl+S（Mac: Cmd+S）でローカル保存する。`
❌ `Windowsキー+左矢印（Mac: Ctrl+左）で画面を左半分に固定する。`
❌ 事前準備に「Windows: タスクバー自動非表示 / Mac: Dock 自動非表示」の両記
❌ 「右クリック（Mac の場合は Ctrl+クリック）」のような但し書き

#### OK パターン

✅ `Ctrl+S でローカル保存する。`
✅ `Windowsキー+左矢印で画面を左半分に固定する。`
✅ 事前準備に「タスクバー自動非表示 ON、集中モード ON」のみ記載
✅ 「右クリック」（両 OS で通じるため）

#### Mac 受講者対応について

Mac 受講者向けの変換は `references/os-conversion-rules.md` に集約され、Step 6.5-P の Mac 版 PDF 生成時にメインClaude が自動で適用する。指示書執筆者は Mac 対応を意識する必要はない。

## 4. 出力フロー（AI/作成者向け）

実践動画の指示書を生成するときは、必ず以下の順で進める：

1. **対象パートの確認**：講座のHow章のうち、どのアクションを実践動画化するか確定
2. **使用ツール・アウトプットの定義**：何を使い、何を残すか
3. **Phase分割**：本編を3〜6個のPhaseに分ける（各Phase 1〜2分目安）
4. **Markdown指示書の作成**：本ガイドラインの§2テンプレートに沿う
5. **CSV指示書の作成**：本ガイドラインの§3に沿う。**操作内容にはプロンプト等の入力文字列を一言一句必ず含める**
6. **整合性チェック**：
   - CSVの全行について「操作」と「台本」が1対1か
   - 操作内容に曖昧表現（「〜について」「〜など」）がないか
   - NG事項（§2-4）が事前準備に書かれているか
   - 想定尺と行数が乖離していないか（目安：1行 = 10〜20秒）
   - **CSV/MD 内に Mac 併記（`（Mac: ...）` 等）が存在しないこと**（§3-6 準拠）
   - **キーボード・ウィンドウ操作が Windows 表記で統一されていること**（§3-6 準拠）

```
  ├─ practice-03-2.csv
  └─ ...
```

- `{講座番号}` はゼロ埋めしない（例: 3, 14, 24）
- 1つの講義に複数の実践動画がある場合は連番（-1, -2, ...）

### 5-2. CSV → Excel 変換仕様（Step 5-C / Step 6 で自動実行）

`practice-{N}.csv` を `/xlsx` スキルで Excel (.xlsx) に変換し、`practice-{N}.xlsx` として併出する。受講者の多くが Excel で開くため、CSV と Excel の両方を提供する。

#### 変換タイミング

| タイミング | 入力 | 出力 |
|---|---|---|
| Step 5-C 完了時（マージ直後） | `practice-{N}.csv` | `practice-{N}.xlsx` |
| Step 6 完了時（51項目レビュー反映後） | `practice-{N}-final.csv` | `practice-{N}-final.xlsx` |

#### 変換仕様

- **シート名**: `操作×台本`
- **1行目**: ヘッダ行（`操作内容` / `台本`）。太字、背景色 `#E8F4FD`、文字色 `#1F4E79`
- **2行目以降**: データ行
- **列幅**: A列（操作内容）= 60、B列（台本）= 80
- **セル内改行**: CSV の `\n`（LF）は Excel の Alt+Enter 相当の改行として保持する
- **セル内文字折り返し**: ON（`wrap_text=True`）
- **フォント**: `Yu Gothic UI`（Windows 標準）。Mac では `Hiragino Sans` にフォールバック
- **行高さ**: データ行は内容に応じて自動調整

#### 変換後のチェック（メインClaude が自動実行）

- CSV の行数と Excel のデータ行数が一致すること
- Excel で開いた際に日本語・記号・セル内改行が文字化けなく表示されること
- ヘッダ行の文字列が CSV と完全一致していること

## 7. PDF手順書の自動生成（Step 6.5-P・Windows 版 + Mac 版の2本出し）

Step 6（51項目レビュー）完了後、`practice-{N}-final.md` と `practice-{N}-final.csv` から**受講者向けPDF手順書**が `scripts/generate_practice_pdf.py` により自動生成される。受講者の OS に応じて以下の2種類を出力する:

- `practice-{N}-guide-win.pdf` — Windows 受講者向け
- `practice-{N}-guide-mac.pdf` — Mac 受講者向け

**重要**:
- PDFは会員（受講者）が手元で実践するための手順書。CSVの「操作内容」「台本」は講師の収録用データであるため、**受講者目線に変換**してから出力する。講師のセリフ・動作指示はそのまま載せない。
- CSV/MD は Windows 表記固定（§3-6）。Mac 受講者向けの OS 変換は PDF 生成時に `references/os-conversion-rules.md` の変換マッピングを適用して自動実施される。
- Mac 版 PDF では事前準備セクションも Mac 向けに再構成される（Dock 自動非表示・Rectangle.app 推奨・おやすみモード等）。

**入出力:**
- 入力: `practice-{N}-final.md` + `practice-{N}-final.csv`（+ Mac 版は `references/os-conversion-rules.md` も参照）
- 出力: `practice-{N}-guide-win.pdf` + `practice-{N}-guide-mac.pdf`（保存先: `output/{project}/05_practice/`）
- PDF 構成（両 OS 共通）: 表紙 → 事前準備チェックリスト → 番号付き操作手順（操作=青背景/ポイント=グレー背景）→ 振り返り
- 変換ルール・詳細仕様は `references/phase2-bulk-pipeline.md` の Step 6.5-P セクション、および `references/os-conversion-rules.md` を参照
