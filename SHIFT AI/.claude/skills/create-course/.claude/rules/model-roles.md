# モデル役割・実行マッピング

## 概要

本スキルは **ハイブリッド体制** で運用する:

- **Step 0/3（リサーチ）**: Gemini API を呼び出し、検索グラウンディングによる探索的リサーチを活用
- **Step 4〜5-B（アウトライン生成・批評・セルフレビュー）**: Claude 4エージェント並列体制で、基準線+会員データに根差した3特化観点（成果直結/離脱防止/クラスター適合）から設計
- **Step 5-C/6/7/8（統合・レビュー・後続工程）**: メインClaude (Opus 4.6) が直接実行

## Claude サブエージェント（Step 4〜5-B でのみ使用）

**Phase 2 の Step 4〜5-B で、メインClaude (Opus 4.6) が4エージェントを並列起動する。**

4エージェントは **基準線（アンカー）** + 会員11,911名の入会動機分析（8クラスター）に根差した **直交する3特化観点** でアウトラインを起案・批評・セルフレビューする。

| エージェント | 役割名 | 担当観点 | 役割・根拠 |
|---|---|---|---|
| baseline-designer | **基準線デザイナー** | 設計GLに忠実な正統派アウトライン | マージ時の基準線（アンカー）。3特化観点がどこまで正統から逸脱しているかの判断材料 |
| outcome-reviewer | **成果直結レビュアー** | 受講後に何ができるようになるか・定量KPI | 88%が実務活用/収益化/不安解消期待、純粋学習は12.4%のみ |
| retention-reviewer | **離脱防止レビュアー** | 最初の成功体験・ハードル・迷子防止 | 22.1%（C3+C4）が不安・未定層で初期離脱リスク最大 |
| segment-fit-reviewer | **クラスター適合レビュアー** | 対象クラスター明確性・制約整合・舞台接地 | 8クラスターは時間制約/価格感度/在宅志向が大きく異なり画一的実践は満足度最大化しない |

4エージェントは独立コンテキストで並列実行され、相互に観点を侵食しない設計。

### 基準線デザイナーの位置づけ

- **性質**: アンカー（基準線）。3特化観点のような「振り切り」ではなく、GLに忠実なバランス型
- **マージ時の役割**: ベース案の有力候補。3特化観点からの差分を上乗せする「土台」として機能
- **批評時の役割**: 他3案がGLからどれだけ逸脱しているかの判断材料を提供
- **独立コンテキスト化の理由**: メインClaude (Opus 4.6) がマージ段階で中立的に編集できるよう、自案を持たない純粋な編集長として機能させるため

## モデル別役割（Step 0/3のリサーチ時）

Step 0/3 では2モデル並列（Claude+Gemini）でリサーチを行う:

- **Claude（メイン）**: 講座設計観点での情報変換重視（GL逆算）
- **Gemini（API）**: 探索的な情報収集・周辺領域の発見・検索グラウンディング重視

## アウトライン工程（Step 4〜5-B）の設計思想

アウトライン生成は **Claude 4エージェント体制** で実行する:

1. **モデル固有の多様性よりも、会員データに根差した観点の直交性が重要**
   - 同じGL・同じリサーチ結果を渡すと、モデル差よりプロンプト差の方が支配的
   - 会員11,911名のクラスター分析から、講座設計で見るべき3つの直交観点が特定できた

2. **基準線デザイナーを独立エージェント化**
   - 旧体制ではメインClaude (Opus 4.6) がGL逆算でアウトラインを生成 + マージも担当 → 自案バイアスが発生
   - 基準線デザイナーを独立コンテキストで生成することで、メインClaude (Opus 4.6) はマージ時に **自案を持たない純粋な編集長** として機能
   - 基準線はマージの「土台」、3特化観点は「上乗せ」という明確な構造が成立

3. **3特化観点はこの会員層に特化して設計**
   - 成果直結（88%の実務期待に対応）
   - 離脱防止（22.1%の不安・未定層に対応）
   - クラスター適合（8クラスター×4カテゴリの多様性に対応）

4. **観点の侵食防止**
   - 各エージェントは自観点に振り切り、他観点は侵食しない
   - マージ段階（Step 5-C）でメインClaude (Opus 4.6) が編集長基準で統合する

5. **コンテキスト効率**
   - 4エージェント（Sonnet 4.6）が各自独立コンテキストで生成 → メインOpusのコンテキスト消費はマージ時の結果読み込みのみ
