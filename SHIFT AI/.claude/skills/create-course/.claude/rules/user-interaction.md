# ユーザー対応・メモリ保存ルール

## ユーザー対応の原則

**初見の人でも最後までついてこれるアシスタントとして振る舞うこと。**

### 処理中の役割名表示（必須）

**各Stepの開始時に、現在の役割名を以下の形式でユーザーに表示すること:**

```
🎭 {役割名}: {実行内容}...
```

役割名の一覧は `model-roles.md` を参照。メインClaude (Opus 4.6) が直接実行する場合はその役割名を、サブエージェントが実行する場合はサブエージェントの役割名を表示する。4エージェント並列起動時は全4役割名を並べて表示する。

### 情報が不足している場合

- ユーザーから必要な情報が得られなかった場合、**やさしく、しかし必ず再度聞く**。曖昧なまま先に進まない
- 催促時は「なぜその情報が必要か」を簡潔に添える
- **代替案・選択肢を必ず提示する**（「例えばこんな感じですか？」「AとBどちらに近いですか？」）
- ユーザーが迷っている場合は、具体例を2〜3個出して選んでもらう

### 成果物の出力時

**ファイルを保存するたびに、保存先のパスをユーザーに提示すること。** ユーザーが成果物の場所を見失わないようにする。

提示フォーマット:
```
📁 保存先: output/{project}/01_outline/{講座名}/detailed-research.md
```

複数ファイルをまとめて保存した場合はフォルダ単位で提示してもよい:
```
📁 保存先: output/{project}/01_outline/{講座名}/
  - outline_claude.md
  - outline_gemini.md
```

### ユーザー確認時

- Phase完了時の確認では、成果物の概要を分かりやすくまとめてから確認を求める
- 「OKですか？」だけでなく、「気になる点があれば何でも言ってください」と添える
- 修正指示が曖昧な場合は、「こういう修正で合っていますか？」と解釈を確認してから実行する

### 環境・依存の不足を検出した場合

**処理フロー中に依存ツールやライブラリの不足を検出した場合、絶対にスキップせず、必ずインストールを誘導すること。**

- コマンド実行時にエラー（`ModuleNotFoundError`、`command not found`、`npm ERR!` 等）が出たら、即座にユーザーに報告する
- 「スキップして続行」は選択肢に含めない。不足を解消してから先に進む
- インストール手順を具体的に提示する（コピペ可能なコマンド付き）
- 可能であれば、その場で自動インストールを実行してよいか確認する
- インストール後、再度コマンドを実行して成功を確認してから先に進む

**提示フォーマット:**
```
⚠️ {ツール名} が見つかりません。このステップで必要です。

インストール方法:
  {具体的なコマンド}

インストールしてよろしいですか？
```

**対象となる依存の例:**
- Python ライブラリ（reportlab, pdfplumber, google-genai 等）
- Node.js パッケージ（textlint, pptxgenjs, playwright 等）
- フォント（Noto Sans JP）
- API キー（GOOGLE_API_KEY）
- Python 仮想環境（.venv）

### 壁打ち・対話時

- ユーザーの発言が短くても意図を汲み取り、確認してから進める
- 専門用語を使う場合は、初回は簡単な説明を添える
- 選択肢を出すときは、各選択肢の違い・メリットを簡潔に説明する
- ユーザーが「分からない」「任せる」と言った場合は、おすすめを提示して理由を添える

## プロジェクト進捗の記録（メモリ保存）

**以下のタイミングでプロジェクトの進捗をメモリに保存すること:**

1. **Phase 1 完了時** — カリキュラム名、セット構成概要、プロジェクト名を保存
2. **各講座の Phase 4 完了時** — 完了した講座名、次に制作すべき講座を保存
3. **セッション終了時（ユーザーが④終了を選択）** — 現在の進捗状態を保存

**保存する情報:**
```
- プロジェクト名: {project-name}
- カリキュラム名: {カリキュラム名}
- コース: {コース名}
- course-structure.md のパス: output/{project}/00_course-design/course-structure.md
- 完了した講座: [{講座名, SET, 種別}]
- 次に制作すべき講座: {講座名}（{SET}・{種別}）
- 最後の作業Phase: Phase {N} Step {M}
```

**メモリの種類:** `project` タイプで保存する。
**ファイル名:** `project_{project-name}.md`

これにより、次回の会話でスキルを起動した際に、前回の続きから再開できる。

## レギュレーション遵守の大原則（最重要）

**このスキルは「品質を守るためのレギュレーション」であり、簡略化・省略・スキップは絶対に禁止。**

### 絶対に簡略化してはいけない工程

| 工程 | レギュレーション |
|---|---|
| Step 0/3 リサーチ | 必ず2モデル並列（Claude+Gemini）で実行。1モデルで代用禁止 |
| Step 4 アウトライン起案 | 必ず4サブエージェント並列（baseline/outcome/retention/segment-fit）で起案。1〜3エージェントで代用禁止 |
| Step 5-A 相互批評 | 必ず4エージェント並列で実施。スキップ禁止 |
| Step 5-B セルフレビュー | 必ず4エージェント並列で実施。スキップ禁止 |
| Step 5-C マージ | 4観点すべてを取り込んで編集長基準でマージ。1観点のみでの採用禁止 |
| Step 6 51項目レビュー | 全該当項目を適用。一部省略禁止 |
| 構造lint（9項目 + P1-P8） | 全項目実施。一部スキップ禁止 |
| textlint | 該当ファイルで必ず実行。0件になるまで繰り返し |
| ユーザー確認（★確認ポイント） | 必ず待つ。勝手に先へ進めるのは禁止 |
| GL ファイルの Read | 必要なGLは必ず Read してから作業開始。推測で進めるの禁止 |

### 簡略化を判断に入れてはいけない理由

- ユーザーが「早くしたい」と言ってもレギュレーションは崩さない
- コンテキストが逼迫しても簡略化ではなくセッション引き継ぎで対応する（`session-handoff.md` 参照）
- 1回の出力短縮より、会員データに根ざした品質の方が優先

### 簡略化を検討せざるを得ない状況で取るべき正しい対応

| 状況 | ❌ 誤った対応 | ✅ 正しい対応 |
|---|---|---|
| コンテキスト残量が厳しい | 3エージェント→1に減らす | `session-handoff.md` に従いセッション引き継ぎ |
| ユーザーから「急いで」と言われた | Step をスキップ | 全工程を守る前提で所要時間を説明し理解を得る |
| 大きなGLファイルを読むのが重い | 読まずに推測 | 必ず Read。容量問題なら引き継ぎ判断 |
| エラーで API 呼び出しが失敗 | 1モデルの結果だけで進める | 再実行。解決しない場合はユーザーに報告 |

## セッション引き継ぎルール（コンテキスト残量対応）

**詳細は `session-handoff.md` を参照。**

要点:
- コンテキスト残量が逼迫しそうな場合、**簡略化せずにセッションを分割** する
- 各 Step の完了時点で `progress.json` を更新（`output/{project}/progress.json`）
- 引き継ぎ宣言時は現 Step を必ず完了させてから停止
- 次セッション起動時に `MEMORY.md` と `progress.json` から自動再開提案

## 注意事項

- 各エージェントはガイドラインファイルを最初に Read で読み込んでから作業を開始する
- コンテキスト最適化: promptファイルには指示のみ書き、大きなファイルは `--merge` で渡す
- ユーザー修正時はチェックリストとの整合性を必ず確認する
- Phase 1 の完了データ（course-structure.md + research-summary.md）は次回会話で再開可能
- Phase 2 は Phase 1 なしでも単体起動可能
- Phase 4 完了後は必ず次アクション選択を提示し、ユーザーの選択を待つ
- セッション終了時は必ずプロジェクト進捗をメモリ + `progress.json` に保存する
