# ツール実践講座 設計ガイドライン

> **出典**: 「ツール実践講座 設計ガイドライン」PDF（社内資料）
> **目的**: 「できる」を設計するための構造。ツール活用講座に特化した設計原則を定義する。

---

## 1. 3つの設計原則

### 原則1: 1講座＝1テーマ

1つの講座で扱うテーマは1つに限定する。テーマが混在する場合は講座を分割し、目的の分散を防ぐ。

**判断基準**: 講座ゴールを一文で言えるか。言い切れない場合はテーマが混在している。

**テーマ混在の弊害**:
- 目的が分散し、焦点がぼやける
- 章構成が肥大化し、学習効率が下がる
- 「機能紹介型」の説明に戻ってしまう
- 受講後に「何ができるようになったか」が曖昧になる

### 原則2: 1ツール＝複数講座

1つのツールを1講座で網羅しない。実務文脈ごとに複数の講座群として設計する。

**推奨構成**: 1ツールにつき3講座前後を基本構成とする（網羅せず、文脈で分ける）

例:
- 基礎実践①: 営業資料改善（既存資料のブラッシュアップ）
- 基礎実践②: 広告改善（キャッチコピー・構成案作成）
- 応用実践: 事業企画整理（複雑な要件定義と構造化）

講座間は「共通の思考プロセス」で接続する: 目的定義 → 条件整理 → 生成実行 → 改善・修正

### 原則3: 目的起点設計

「機能」から入らない。常に「やりたいこと（目的）」から入り、その手段として技術を学ぶ。

**章内の基本フロー**:
1. やりたいことを提示する
2. 目的を具体化する
3. 必要な技術を抽出する
4. 技術を学ぶ
5. 再び目的に戻り、成果を確認

章の終わりは「何ができるようになったか」で締める。

## 2. 設計プロセス全体像（抽象 → 構造 → 具体）

### STEP 1: 講座ゴール設計（抽象）

#### ペルソナ定義
- 経験値（未経験/初級/中級）
- 現在の課題
- 活用目的
- 想定活用シーン
- ※誰のどんな課題を解決するのかを明確にする

#### 行動ゴールの設定
- 受講者が「○○ツールを用いて△△を実行/作成できる」状態になる
- **NGワード**: 「理解する」「知る」は使用しない
- 行動動詞で記述する

#### Before → After
- 感情面の変化
- 知識面の変化
- スキル面の変化
- 行動面の変化
- ※ゴールは関係者間で合意形成する

### STEP 2: 小構成（ブロック）設計（構造）

#### ロール設計

**ロールの定義項目**:
- 業界
- 会社規模
- 職種・立場
- 直近のミッション
- 成果物
- 期限

**重要**: ロールは演出ではない。意思決定の前提条件である。

**ロール設計の基準**:
- 明日あり得る実務である
- 成果物が明確である
- ツールを使う必然性がある
- 受講者が想像できる立場である

**原則**: 1講座＝1ロール。基本実践ブロック内でロールを変えない。応用実践でのみ文脈変更を行う。

#### 標準ブロック構成

| ブロック | 名称 | 目的 | 章数目安 |
|---|---|---|---|
| Block 0 | 導入 | ロール提示、活用イメージ形成、心理的ハードル低下、最初の成功体験 | 1章 |
| Block 1 | 基本実践 | ロール内で基本的な扱い方を体験的に理解 | 3〜5章 |
| Block 2 | 精度向上 | ツールを意図的に扱える状態へ（指示具体化/条件追加/工程分割） | 2〜3章 |
| Block 3 | 応用実践 | 基本を異なる文脈で再現する（シナリオ変更/難易度向上） | 2〜4章 |

**推奨合計**: 8〜12章

#### 導入ブロック — 構成フロー

1. ロール提示（状況・ミッション共有）
2. 未来体験（完成例提示）
3. 最小限の概要説明
4. 環境構築
5. 初回成功体験

導入は短時間で設計し、すぐに「できた」を感じさせる。

#### 基本実践ブロック — テーマ選定プロセス

1. 実務用途を洗い出す（目安20以上）
2. 用途をクラスタリングする
3. ロールに適合する用途を選定
4. 基準で絞り込む（汎用性・再現性）

テーマ数: 3〜5テーマ。すべて同一ロール内で完結させる。

**各章の基本構造（1章1テーマ）**:
1. ミッション提示
2. 実行
3. 改善要素提示
4. 再実行
5. 振り返り

#### 精度向上ブロック

**扱う観点**:
- 指示の具体化
- 出力形式の指定
- 条件の追加
- 工程の分割

#### 応用実践ブロック

**各章の構造（自走力を高める）**:
実務提示 → 自力実行 → ヒント → 再実行

**テーマ選定**: 実務シナリオを洗い出し、難易度順に整理する

### STEP 3: 各章アウトライン設計（具体）

各章は以下の型で統一する:

| フェーズ | 内容 | ポイント |
|---|---|---|
| Why | なぜこの章が必要か | 目的が曖昧なまま技術に入らない。必ず「やりたいこと」を先に提示 |
| What | 扱うテーマと範囲 | 1章で扱う機能は1〜2個まで。「広く浅く」ではなく「狭く深く」 |
| How | 実践パート | 説明よりも体験を優先。「理解した」ではなく「できた」状態を作る |
| Wrap | 章まとめ | 「学んだ機能」ではなく「できるようになったこと」で締める |

## 3. NG例

### NG①: 機能を主語にしない

- ×「ロゴ生成機能」→ ○「ロゴを作ってみる」
- ×「背景削除機能」→ ○「ポスター用に写真を整える」
- ×「モックアップ作成方法」→ ○「完成ロゴを商品に展開する」

### NG②: 1章に詰め込まない

1章につき目的は1つ。扱う機能は1〜2個まで。
章の最後は「この章で○○ができるようになった」と言える状態にする。

### NG③④: 操作説明から入らない

唐突な操作説明（「背景削除の方法を説明します」）ではなく、課題・文脈を先に提示する:
1. どんな状況か
2. 何を実現したいのか
3. なぜ困っているのか

操作は、その課題を解決するために初めて登場させる。

### NG⑤: 技術が独立して存在しない

「良いプロンプトの条件は〜です」のように文脈なく抽象論で終わらない。
技術は必ず具体的な目的の中で扱う（例: 「ロゴが崩れる」→「プロンプト修正」）。

## 4. レビュー時の確認事項

- [ ] 各章で実現する目的は明確か
- [ ] 章タイトルは「やりたいこと」になっているか
- [ ] 1章で扱う機能は1〜2個に収まっているか
- [ ] 実務は文脈として機能しているか
- [ ] 章の終わりに「できるようになったこと」が明確か

---

## 5. GRAND RULE

**「理解した」ではなく「できた」を作る。**

ツールはあくまで手段。受講者が「自分の業務で使える」と確信できる状態を目指す。

---

## 6. 設計原則の比較（機能網羅 vs 実践密度）

| 観点 | 機能の網羅（辞書型）❌ | 実践密度（武器型）✅ |
|---|---|---|
| 方針 | 「あれもこれも」と機能を詰め込む | 「狭く深く」設計する |
| 結果 | 知っている機能100個、使える機能0個 | 確実に成果を出せる5個の機能を徹底習得 |
| 推奨 | — | **推奨** |

## 7. 教材クリエイター提出物一覧

1. ペルソナ定義（誰に向けた講座か）
2. 行動ゴール（Before / After の状態定義）
3. ロール設定（業界・職種・ミッションの具体化）
4. 講座タイトル案（受講者が自分事化できるタイトル）
5. ブロック構成案（導入・基本・精度・応用の配分）
6. 章リスト（全章のタイトル一覧）
7. 各章アウトライン（Why / What / How / Wrap の詳細）

**全ての設計承認が完了してから、スライド・動画の制作に入る。**
