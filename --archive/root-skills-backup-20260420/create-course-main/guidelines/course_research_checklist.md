# コースリサーチ チェックリスト

> **目的**: カリキュラム設計に必要な全情報を一括収集し、後続の教養講座・実務講座制作にも使える素材を整える。
> 収集した情報はそのまま並べるのではなく、カリキュラム設計GL・教養GL・実務GLの各要素に変換して整理すること。
> （参照: `0_search/report_v1.txt`, `guidline/教養と実務_カリキュラム設計ガイドライン.md`, `guidline/教養_講座設計ガイドライン.md`, `guidline/実務_講座設計ガイドライン.md`）

---

## 使い方

1. カリキュラム名（テーマ）を入力として受け取る
2. STEP① で調査準備を行う
3. STEP② でカリキュラム設計用の情報を収集する
4. STEP③ で教養講座素材を収集する
5. STEP④ で実務講座素材を収集する
6. STEP⑤ で市場・競合情報を収集する
7. STEP⑥ で収集情報をカリキュラム設計観点へ変換する
8. STEP⑦ でレポート品質を最終確認する

---

## STEP① 調査準備

### 1-A. カリキュラムテーマの理解

- [ ] カリキュラムが属するコースの文脈を把握した（例: 業務効率化コース > リサーチ）
- [ ] このカリキュラムが受講者のどんな課題を解決するものかを仮説として設定した
- [ ] 調査の目的を「カリキュラム設計 + 後続の全講座制作に必要な全情報の収集」と明文化した

### 1-B. ターゲットクラスターの特定

- [ ] クラスター分析サマリー（`ref/cluster-summary.md`）を読み込んだ
- [ ] このカリキュラムの主対象クラスター（1〜3つ）を特定した
- [ ] 各ターゲットクラスターの入会動機・課題・目標金額を把握した
- [ ] ターゲットクラスターの特徴に基づき検索方針を調整した

### 1-C. 調査論点の設定

- [ ] このカリキュラムのゴール候補を仮説として設定した
- [ ] 含まれる実務タスクの範囲を仮説として設定した
- [ ] 競合となる既存講座・書籍の存在を確認する方針を立てた

---

## STEP② カリキュラム設計用の調査（→ カリキュラム設計GL §5 に直結）

### 2-A. ゴール候補の調査

- [ ] この分野で求められるスキル・能力を10以上リストアップした
- [ ] この分野の成果物（受講者が最終的に作れるもの）を5つ以上特定した
- [ ] 「このカリキュラムを修了すると○○ができる」の候補を3つ以上作成した
- [ ] ターゲットクラスターの目標（収入・キャリア・効率化）とゴールの整合性を確認した

### 2-B. 実務タスクの網羅的洗い出し

- [ ] この分野で実際に発生する業務タスクを20以上リストアップした
- [ ] 各タスクがAIで効率化できるかどうかを判定した
- [ ] 各タスクで使用するツール・機能（1〜2個）を特定した
- [ ] 各タスクの難易度（初級/中級/上級）を判定した
- [ ] タスクを「やりたいこと」形式のタイトルで表現した（機能名ではなく目的起点）
- [ ] 手作業タスクとAI活用タスクの両方を含めた

### 2-C. 教養テーマの洗い出し

- [ ] 各実務タスクの前提となる概念・知識を特定した
- [ ] この分野の体系的な知識構造（主要フレームワーク）を調査した
- [ ] 概念・知識と実務タスクの対応関係（どの知識がどのタスクに必要か）を整理した
- [ ] 教養テーマが実務に直結する「Just-in-Time」の知識になっているか確認した

### 2-D. 難易度・順序の調査

- [ ] 初心者→上級者の学習パスを調査した（書籍の目次順、カリキュラム構成等）
- [ ] 前提知識の依存関係を特定した（Aを知らないとBができない、等）
- [ ] Lv分け（2〜4段階）の候補を設定した
- [ ] 各Lvのテーマ名候補（例: 「個人プレイヤー」「仕組み化・ツール化」）を作成した

---

## STEP③ 教養講座素材の収集（→ 教養GL に直結）

### 3-A. 受講者の悩み・課題（→ 教養GL: Whyパート設計）

- [ ] Q&Aサイト（Yahoo!知恵袋・Quora等）から悩みを5件以上収集した
- [ ] SNS（X・note等）から生の声を5件以上収集した
- [ ] 受講者が使う言葉・表現をそのまま記録した（講師目線に翻訳しない）
- [ ] 受講者のよくある誤解・思い込みを3つ以上収集した
- [ ] 「学ばなかった場合のリスク」の事例を3つ以上収集した

### 3-B. 概念・フレームワーク・思考の型（→ 教養GL: What/Howパート設計）

- [ ] この分野の主要な概念・専門用語を10以上リストアップした
- [ ] 各概念の平易な言い換え（専門用語→日常語）を準備した
- [ ] 専門家が使う思考プロセス（判断基準・分析手順）を3つ以上調査した
- [ ] よくある失敗パターンとその回避策を5つ以上収集した
- [ ] 身近な例え話に使える比喩・アナロジー候補を収集した

### 3-C. 書籍・講座の構成参照（→ 教養テーマの体系把握）

- [ ] 同分野の書籍の目次構成を3冊以上確認した
- [ ] 同分野の既存講座のカリキュラム構成を3件以上確認した
- [ ] 教養的知識の「教える順番」の標準パターンを把握した

---

## STEP④ 実務講座素材の収集（→ 実務GL に直結）

### 4-A. ペルソナ候補（→ 実務GL: ペルソナ4項目）

- [ ] この分野のツール・スキルを使う実務者の職種・役職を5パターン以上特定した
- [ ] 各パターンの日常業務を調査した
- [ ] 各パターンの現在の課題（AI未活用時の困りごと）を調査した
- [ ] 各パターンのAI利用経験レベルを推定した
- [ ] ターゲットクラスターとの適合性が高いペルソナを推定した

### 4-B. Before/After数値（→ 実務GL: Before/After設計）

- [ ] 主要な実務タスク3つ以上について、AI導入前の所要時間・コストを調査した
- [ ] AI導入後の所要時間・コストを調査した（事例・記事・レポートから）
- [ ] 「メール1通15分→3分」のような具体的な定量変化を記録した

### 4-C. サンプルプロンプト素材（→ 実務GL: Howパート設計）

- [ ] 実際の業務で使われているプロンプト事例を5件以上収集した
- [ ] 各プロンプトの出典（記事・コミュニティ・公式ガイド）を記録した
- [ ] プロンプトの用途（何のタスクに使うか）を分類した

### 4-D. 使用ツール候補（→ 実務GL: Whatパート設計）

- [ ] この分野で使えるAIツール・Webサービスをリストアップした
- [ ] 各ツールの特徴・適したタスクを整理した
- [ ] 無料/有料、初心者向け/上級者向けを分類した

### 4-E. コース文脈の判断材料（→ 実務GL: §4 コース文脈）

- [ ] このカリキュラムが「副業」「業務効率化」のどちらに適するか判断する材料を収集した
- [ ] 受講者の動機（副業収入 vs 業務時短）に関する調査データを確認した
- [ ] 推奨コース文脈と判断理由を整理した

---

## STEP⑤ 市場・競合調査（既存6観点を継承）

### 5-A. 市場動向

- [ ] テーマが社会的・業界的に拡大しているかを確認した
- [ ] どの層で重要性が増しているかを確認した
- [ ] 一時的な話題性か、継続的な教育価値があるかを見極めた

### 5-B. 企業ニーズ

- [ ] 求人情報からテーマに関連するスキル要件を3件以上確認した
- [ ] 企業の導入事例・活用事例を3件以上収集した

### 5-C. 競合講座・類似コンテンツ

- [ ] 同テーマの既存講座・セミナーを5件以上リストアップした
- [ ] 各競合の対象者・価格帯・カバー範囲を記録した
- [ ] 競合が不足している領域（差別化余地）を特定した

### 5-D. 技術・ツールの変化

- [ ] テーマに関連する最新ニュース・トレンドを3件以上収集した
- [ ] 概念として長く使える内容か、頻繁な更新が必要かを見極めた

---

## STEP⑥ カリキュラム設計観点への変換

### 6-A. カリキュラム設計GL用の変換

- [ ] 実務タスク一覧を「セット」の単位でグルーピングした（関連タスクをまとめる）
- [ ] 各セットに必要な教養テーマを紐付けた（パターンA/B/C）
- [ ] セットをLv（難易度段階）でグルーピングした
- [ ] 学習順序（簡単→難しい、基礎→応用）を検討した

### 6-B. 教養GL用の変換

- [ ] 受講者の悩み → 各教養講座のWhyパートの素材として整理した
- [ ] 概念・フレームワーク → 各教養講座のWhat/Howパートの素材として整理した
- [ ] 失敗パターン → Howパートの「やってはいけない例」として整理した

### 6-C. 実務GL用の変換

- [ ] ペルソナ候補をターゲットクラスターと照合し最適パターンを推定した
- [ ] 各実務タスクを「やりたいこと」形式の動画タイトル候補に変換した
- [ ] Before/After数値を定量的な受講効果として整理した
- [ ] サンプルプロンプト素材をHowパートで使える形式に整理した

### 6-D. 4層分離（事実と解釈の分離）

収集・変換した情報を以下の4層に分類する:

- [ ] **Layer 1: 事実情報（出典付き）** — 確認済みのファクト。出典URL・時点を必ず記載
- [ ] **Layer 2: 複数ソースで一致した論点** — 2つ以上の独立ソースで確認できた情報
- [ ] **Layer 3: 解釈・仮説（要検証）** — 単一ソースまたは推論による情報。教材に使う場合は裏取り必要
- [ ] **Layer 4: 使用禁止情報** — 以下に該当する情報を明示的に除外リストに入れる:
  - エビデンス不足の主張
  - トレンド依存で短命な話
  - 教材目的から逸れる論点
  - 初学者には過剰な専門論
- [ ] 数値データには出典と時点を明記した
- [ ] Layer 3 の情報が Layer 1 として扱われていないか確認した

---

## STEP⑦ レポート品質チェック（最終確認）

### カリキュラム設計用の品質

- [ ] ゴール候補が具体的で、ターゲットクラスターのニーズに合致している
- [ ] 実務タスクが20以上洗い出され、難易度・ツールが整理されている
- [ ] 教養テーマと実務タスクの対応関係が明確である
- [ ] Lv分け・学習順序の根拠がある

### 教養講座素材の品質

- [ ] 受講者の悩みが「生の声」レベルで具体的である（講師目線の抽象化ではない）
- [ ] 概念・フレームワークがこの分野を網羅している
- [ ] 失敗パターンが具体的で、教養講座のHowパートにそのまま使える

### 実務講座素材の品質

- [ ] ペルソナが5パターン以上、4項目（職種・日常業務・課題・AI経験）で具体的
- [ ] Before/After に定量データが3テーマ以上で含まれている
- [ ] サンプルプロンプト素材が5件以上で出典付き

### 全体の品質

- [ ] ターゲットクラスターが明記され、調査全体がクラスターニーズに基づいている
- [ ] 「一般的な学習者」ではなく「自社会員の具体的ニーズ」で語れている
- [ ] 事実と解釈が分かれている
- [ ] 参考ソースが記録されている
