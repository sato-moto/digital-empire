# 品質レビュー基準

## Step 6: 51項目レビューの適用ルール

**メインClaude（Opus 4.6）が直接実行する。サブエージェントには委任しない。**

`references/review-checklist.md` の51項目チェックリストでアウトラインをレビューし、改善版を出力する。

### 講座タイプ別の適用範囲

| 講座タイプ | 適用範囲 |
|---|---|
| **教養** | 共通(#1〜37) + 教養系追加(#38〜39) + 工程整合(#47〜51) |
| **実務** | 共通(#1〜37) + 実務講座追加(#40〜46) + 工程整合(#47〜51) |
| **従来** | 共通(#1〜37) + 工程整合(#47〜51)。講座内容に応じて教養追加(#38〜39) or 実務追加(#40〜46) を適用 |
| **教養（SET）** | 共通(#1〜37) + 教養系追加(#38〜39) + 工程整合(#47〜51) + セット整合(#52〜57) |
| **実務（SET）** | 共通(#1〜37) + 実務講座追加(#40〜46) + 工程整合(#47〜51) + セット整合(#52〜57) + **収録指示書チェック(P1〜P8)** |

### レビューパイプライン

1. `outline.md` と `references/review-checklist.md` を Read
2. 講座タイプ（教養/実務/従来）を判定し、適用するチェック範囲を決定
3. 51項目チェックリストで1項目ずつ評価（✅/⚠️/❌）
4. ⚠️❌項目に「何が問題か・なぜ問題か・どう直すか」の3点セットを記述
5. 改善を反映 → `outline-final.md`
6. textlint 実行: `cd "{skill_root}" && npx textlint "{out}/outline-final.md"`
7. レビュー結果を `review-report.md` に保存
8. ユーザーに提示して確認（修正指示があればチェックリスト影響を確認して反映）

**成果物**: `outline-final.md` + `review-report.md`

## 構造lint（9項目）

textlint の前に必ず実行する:

| # | チェック項目 |
|---|---|
| 1 | 同階層で抽象度が揃っているか |
| 2 | 同一章に重複論点がないか |
| 3 | 各章がゴールに紐づいているか |
| 4 | 前提知識なしで理解できる順序か |
| 5 | Why/What/How/Wrap が全章に存在するか |
| 6 | 章間接続（前章結論→疑問→今章必要性）があるか |
| 7 | 1章のテーマが1〜2個に収まっているか |
| 8 | ミニ台本が400字以上あるか |
| 9 | 固有名詞（企業名・人名・出典元サービス名）が本文に残っていないか |
| 10 | SET単位の場合、ファイル名が outline_setN.md / textmaterial_setN.md に準拠しているか |

## セット整合チェック（SET単位制作時のみ適用）

SET単位で制作する場合（course-structure.md に「標準一貫ケース」が定義済み）、以下を追加でチェックする:

| # | チェック項目 | 確認内容 |
|---|---|---|
| 52 | 一貫ケースデータ埋込 | 一貫ケースのデータが全プロンプト例に正しく埋め込まれているか。汎用プレースホルダー `[あなたの課題]` が残っていないか |
| 53 | 教養How別事例 | 教養講座のHow具体例が一貫ケースと別業界・別問題パターンを使用しているか |
| 54 | スキルが先→AI活用が後 | 各実務講座でフレームワーク解説→AI操作の順序が守られているか |
| 55 | 2ブロック構成分離 | 実務講座が outline_setN.md（動画デモ）と textmaterial_setN.md（テキスト教材）に分離されているか |
| 56 | テキスト教材品質 | テキスト教材のプロンプトが一貫ケースデータ入りで即実行可能な状態か |
| 57 | 問題解決ストーリー接続 | 前SETの結論を受けて今SETが始まっているか。前SETの成果物が入力になっているか |

## 収録指示書チェック（実務講座・SET単位制作時のみ適用）

`practice_video_guideline.md` §6 に基づく整合性チェック:

| # | チェック項目 | 確認内容 |
|---|---|---|
| P1 | 操作×台本 1対1 | CSVの全行で「操作内容」と「台本」が対になっているか |
| P2 | 曖昧表現なし | 操作内容に「〜について」「〜など」「〜を入力する」（具体文字列なし）がないか |
| P3 | プロンプト全文記載 | 入力文字列がコピペ可能な一言一句で記載されているか |
| P4 | NG事項記載 | 事前準備にデスクトップバー/通知/個人情報等のNG事項が書かれているか |
| P5 | 尺と行数の整合 | 想定尺と行数が乖離していないか（目安: 1行 = 10〜20秒） |
| P6 | 敬語統一 | 全文末が「です・ます／ません・ました／ましょう」で終わっているか |
| P7 | 必須構成要素 | オープニング（実践切り出し・前講座接続・ゴール宣言）とエンディング（振り返り・ねぎらい・次への接続）が揃っているか |
| P8 | 一貫ケース埋込 | プロンプト例に一貫ケースのデータが正しく埋め込まれているか |

## Claude 編集長基準（Step 5-C + Step 0統合 共通）

メインClaude が統合・選択する際の明文化された優先順位:

**評価優先順位（上から順に優先）:**
1. 受講者ゴールに直結しているか
2. 前後講座の文脈に自然につながるか
3. 章立ての抽象度が揃っているか
4. 重複と抜け漏れがないか
5. 教えた時に理解順序が破綻しないか
6. 実務に落とした時に行動に変わるか
7. 表現が明瞭か

**許容する欠点（後工程で改善可能）:**
- 表現の洗練度
- ミニ台本の文量

**許容しない欠点（この段階で解消必須）:**
- ゴールとの不整合
- 学習順序の破綻
- 前後講座との断絶

## 固有名詞の除去ルール（Step 4 生成時に適用）

アウトラインは受講者向けの教材であり、リサーチの出典情報をそのまま含めてはならない。
- ❌ 禁止: 企業名（「○○社の調査によると」）、人名（「△△氏が提唱する」）、サービス名（「□□ツールでは」）
- ✅ 許可: 一般化した表現（「ある調査によると」「専門家の間では」「主要なBIツールでは」）
- ✅ 例外: 講座で使用するツール名（ChatGPT, Excel等）はカリキュラム設計上必要なため許可
