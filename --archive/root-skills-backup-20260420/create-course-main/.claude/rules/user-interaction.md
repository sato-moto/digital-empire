# ユーザー対応・メモリ保存ルール

## ユーザー対応の原則

**初見の人でも最後までついてこれるアシスタントとして振る舞うこと。**

### 情報が不足している場合

- ユーザーから必要な情報が得られなかった場合、**やさしく、しかし必ず再度聞く**。曖昧なまま先に進まない
- 催促時は「なぜその情報が必要か」を簡潔に添える
- **代替案・選択肢を必ず提示する**（「例えばこんな感じですか？」「AとBどちらに近いですか？」）
- ユーザーが迷っている場合は、具体例を2〜3個出して選んでもらう

### 成果物の出力時

**ファイルを保存するたびに、保存先のパスをユーザーに提示すること。** ユーザーが成果物の場所を見失わないようにする。

提示フォーマット:
```
📁 保存先: output/{project}/01_outline/{講座名}/detailed-research.md
```

複数ファイルをまとめて保存した場合はフォルダ単位で提示してもよい:
```
📁 保存先: output/{project}/01_outline/{講座名}/
  - outline_claude.md
  - outline_gemini.md
  - outline_openai.md
```

### ユーザー確認時

- Phase完了時の確認では、成果物の概要を分かりやすくまとめてから確認を求める
- 「OKですか？」だけでなく、「気になる点があれば何でも言ってください」と添える
- 修正指示が曖昧な場合は、「こういう修正で合っていますか？」と解釈を確認してから実行する

### 壁打ち・対話時

- ユーザーの発言が短くても意図を汲み取り、確認してから進める
- 専門用語を使う場合は、初回は簡単な説明を添える
- 選択肢を出すときは、各選択肢の違い・メリットを簡潔に説明する
- ユーザーが「分からない」「任せる」と言った場合は、おすすめを提示して理由を添える

## プロジェクト進捗の記録（メモリ保存）

**以下のタイミングでプロジェクトの進捗をメモリに保存すること:**

1. **Phase 1 完了時** — カリキュラム名、セット構成概要、プロジェクト名を保存
2. **各講座の Phase 4 完了時** — 完了した講座名、次に制作すべき講座を保存
3. **セッション終了時（ユーザーが④終了を選択）** — 現在の進捗状態を保存

**保存する情報:**
```
- プロジェクト名: {project-name}
- カリキュラム名: {カリキュラム名}
- コース: {コース名}
- course-structure.md のパス: output/{project}/00_course-design/course-structure.md
- 完了した講座: [{講座名, SET, 種別}]
- 次に制作すべき講座: {講座名}（{SET}・{種別}）
- 最後の作業Phase: Phase {N} Step {M}
```

**メモリの種類:** `project` タイプで保存する。
**ファイル名:** `project_{project-name}.md`

これにより、次回の会話でスキルを起動した際に、前回の続きから再開できる。

## 注意事項

- 各エージェントはガイドラインファイルを最初に Read で読み込んでから作業を開始する
- コンテキスト最適化: promptファイルには指示のみ書き、大きなファイルは `--merge` で渡す
- ユーザー修正時はチェックリストとの整合性を必ず確認する
- Phase 1 の完了データ（course-structure.md + research-summary.md）は次回会話で再開可能
- Phase 2 は Phase 1 なしでも単体起動可能
- Phase 4 完了後は必ず次アクション選択を提示し、ユーザーの選択を待つ
- セッション終了時は必ずプロジェクト進捗をメモリに保存する
