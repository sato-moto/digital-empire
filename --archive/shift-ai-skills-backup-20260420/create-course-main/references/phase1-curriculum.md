# Phase 1: カリキュラム設計 — 詳細仕様

カリキュラム（複数のセット × 複数の動画講義）の全体構成を設計する。
ユーザーとの対話を通じてブラッシュアップし、確定させる。

## Step 0: カリキュラムリサーチ

カリキュラムテーマについて、カリキュラム設計 + 後続の全講座制作に必要な情報を一括収集する。

**パイプライン:**
1. メインClaude が生成プロンプトを作成 → `{out}/prompt.md`（モデル別役割指示を含む）
2. **メインClaude（Opus 4.6）** が Claude版リサーチを生成 → `{out}/ver_claude.md`
3. **並行して** api-caller エージェントが Gemini + OpenAI を API 呼び出し → `ver_gemini.md`, `ver_openai.md`
4. **各版ファクトチェック** → 3版それぞれに WebSearch で裏取り（統合前に品質担保）
5. **各版セルフチェック** → 3版それぞれに14項目チェック + 改善（統合前に品質担保）
6. **メインClaude** が品質担保済みの3版を統合 → `research-summary.md`（統合判断基準9項目 + Claude編集長基準）

**research-summary.md の4層構造:**
```
## Layer 1: 事実情報（出典付き）
確認済みのファクト。出典URL・時点を必ず記載。

## Layer 2: 複数ソースで一致した論点
2つ以上の独立ソースで確認できた情報。

## Layer 3: 解釈・仮説（要検証）
単一ソースまたは推論による情報。教材に使う場合は裏取り必要。

## Layer 4: 使用禁止情報
- エビデンス不足の主張
- トレンド依存で短命な話
- 教材目的から逸れる論点
- 初学者には過剰な専門論
```

**api-caller エージェントに渡すもの:**
- カリキュラム名（テーマ）
- 出力先: `output/{project}/00_course-design/`
- 生成プロンプト（`prompt.md`）— モデル別役割指示を含む

**成果物**: `research-summary.md`（4層構造）
- カリキュラム設計素材（ゴール候補・実務タスク一覧・教養テーマ一覧・Lv分け候補）
- 教養講座素材（受講者の悩み・概念/フレームワーク・失敗パターン）
- 実務講座素材（ペルソナ候補・Before/After数値・サンプルプロンプト・ツール候補）
- 市場・競合分析

## Step 1: カリキュラム構成提案 + 壁打ち

research-summary.md をもとに、メインエージェント（Claude）がカリキュラムの全体構成を **一括で提案** する。

**提案フォーマット:**

```markdown
# {カリキュラム名} 全セット構成図
{一行キャッチコピー}

## ゴール
このカリキュラムを修了すると:
- {ゴール1}
- {ゴール2}

## Lv 1: {レベル名}

### SET 1: {カリキュラム全体像のセット名}
- 【教養】{カリキュラム全体の意義・全体像を伝える講座タイトル}

### SET 2: {セット名}
- 【教養】{講座タイトル}
- 【実務】{講座タイトル} [{使用ツール}]

### SET 3: {セット名}
- 【教養】{講座タイトル}
- 【実務】{講座タイトル} [{ツール}]
- 【実務】{講座タイトル} [{ツール}]

## Lv 2: {レベル名}

### SET 3: {セット名}
...
```

**セットのパターン（カリキュラム設計GL §4 準拠）:**
- **パターンA**: 教養のみ（導入・マインドセット）— **SET 1 は必ずこのパターン**（カリキュラム全体像・意義を伝える教養1本のみ）
- **パターンB**: 教養1本 + 実務1本
- **パターンC**: 教養1本 + 実務複数本

**提案時の設計プロセス（カリキュラム設計GL §5 準拠）:**
1. ゴール設定 — 修了後に「何ができるようになっているか」
2. 実務タスク洗い出し — ゴールから逆算した必要タスク
3. 教養→実務のセット化 — 各タスクに必要な前提知識を紐付け
4. 学習順序 + Lv分け — 簡単→難しい、基礎→応用

**壁打ち:**
- ユーザーが修正指示を出したら反映して再提示
- 「SET追加して」「Lv分けを変えて」「この実務を別のセットに移して」等
- ユーザーが「完成」と宣言するまで繰り返す

**カリキュラム設計チェックリスト（カリキュラム設計GL §7 準拠）:**
提案時・完成時に以下を確認する:
- [ ] カリキュラムのゴールは具体的で、受講者がメリットを感じられるか？
- [ ] 動画講義群は「教養→実務」のセットで構成されているか？
- [ ] 各「実務」の前に、必要な「教養」が適切に配置されているか？
- [ ] 実務動画だけで、カリキュラムのゴール達成に必要なスキルが身につくか？
- [ ] 標準一貫ケースが定義されているか？（実務講座を含むカリキュラムの場合）
- [ ] 問題解決ストーリーがSET間で連続しているか？（実務講座を含むカリキュラムの場合）
- [ ] 教養How事例が一貫ケースと別業界から選ばれているか？（実務講座を含むカリキュラムの場合）

## Step 2: 制作対象の選択

**重要**: アウトラインは講座単位（1講座 = 1アウトライン）で設計する。
**重要**: 対象範囲の全講座のアウトラインを Phase 2 で作り切ってから Phase 3 に進む。1講座ごとに Phase 3 に流れないこと。

ユーザーに「どの範囲の講座を作るか」を選択させる:

```
Phase 1 が完了しました！
Phase 2 では、選んだ範囲の講座のアウトラインを全て作り切ります。
全アウトライン完成後に Phase 3（スライド構成）へ進みます。

どの範囲で進めますか？

① おすすめ: Lv.1 の全講座を上から順に作る（全SET単位）
  → course-structure.md の Lv.1 の全SETを、上から順にアウトライン設計します。

② 範囲を指定して制作する（SET単位）
  → SET単位で範囲を指定できます。
```

- **① を選んだ場合**: course-structure.md の Lv.1 の全未着手SETを上から順に Phase 2 で制作する。全講座のアウトライン完成後、まとめて Phase 3 へ進む確認を行う。
- **② を選んだ場合**: course-structure.md の全SET一覧（ステータス付き）を表示し、ユーザーにSETを指定させる。

選択された講座の種別（教養/実務/従来）に応じて、Phase 2 で使用するガイドラインとパイプラインを自動で判定する:

**SET選択後の前処理（Step 3 の前に実行）:**
1. `guidelines/実務_講座設計ガイドライン_2.md` を Read
2. course-structure.md から対象SETの情報を抽出
3. SET内設計方針を確認・補完:
   - 一貫ケースが定義済みか → 未定義なら設計
   - 問題解決ストーリーのこのSETの位置づけ
   - 教養How用別事例の割当
   - 使用ツール
4. ユーザーに確認後、SET内の各講座をcourse-structure.mdの順で Step 3 へ
- **【教養】** → `guidelines/教養_講座設計ガイドライン.md` → 一括パイプライン（Step 3〜6）
- **【実務】** → `guidelines/実務_講座設計ガイドライン.md` → 一括パイプライン（Step 3〜6）
- **【従来】** → `guidelines/outline_v1.md` → 段階パイプライン（Step 3 + Step 4-T1〜T4 + Step 5-T + Step 6）
